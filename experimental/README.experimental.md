# 实验性分支：LoKr 支持（EXPERIMENTAL）

> ⚠️ 这是实验性功能，独立于正式发布的两个节点，未经长期验证。正式使用请用发布版节点。

## 这是什么

正式版的两个节点只支持普通 LoRA（`lora_down`/`lora_up` 结构）。
本实验分支额外支持 **LoKr**（一种 LoRA 的变体，权重存成 `lokr_w1`/`lokr_w2` 两块，靠特殊矩阵乘法拼起来）。

包含两个实验节点（菜单分类 `loaders/experimental`）：
- **Anima LoKr Block Weight [实验性]** — 运行时分层加载
- **Anima LoKr Block Weight Export [实验性]** — 烘焙导出

它们也兼容普通 LoRA（自动检测格式），但请仍把它当实验品。

> 节点标题里的 `[实验性] / [Experimental]` 后缀会跟着 ComfyUI 的界面语言自动切换
> （中文界面显示 `[实验性]`，英文界面显示 `[Experimental]`）。
> 如果你手动给节点改过名字，则保留你的命名、不会被覆盖。

**本次更新**：两个实验节点已对齐发布版 V2 的参数体系——grouped 四段
（seg_motion / proportion / core / detail + 区间可改）、per_block 28 滑块、子模块系数
（w_self_attn / cross_attn / mlp / adaln 四个子模块系数），并**共享发布版同一界面面板**（含影响力染色）。
即不再是旧的 shallow/middle/deep 三档 + 文本框。

## 为什么 LoKr 要单独处理

普通 LoRA 每个模块是 `lora_down` + `lora_up` 两块；
LoKr 不一样，每个模块是 `lokr_w1` + `lokr_w2` 两块，最终效果是这两块做"克罗内克积"
（一种特殊的矩阵相乘，记作 ⊗）拼出来的。

有一条数学性质（已用真实文件验证）：
```
(系数 c × lokr_w1) ⊗ lokr_w2  等于  c × (lokr_w1 ⊗ lokr_w2)
```
意思是：只要把缩放系数乘到 **lokr_w1 这一块**，效果就等于整体缩放了 c 倍。
**绝对不能两块都乘**——那样会变成 c 的平方，缩放就错了。

### 影响力染色（LoKr 专属）

发布版给普通 LoRA 算"影响力"的办法，是把每个 block 的权重数值大小（数学上叫 L2 范数，
可以理解成"这块权重整体有多大劲")加起来。

LoKr 不能直接拿 `lokr_w1` 一块的大小来算（那样漏掉了 `lokr_w2`，会导致 block 之间排序排错）。
正确的办法是算"拼起来之后的等效权重"有多大。这里又有一条省事的性质：
```
两块拼起来的整体大小  等于  第一块的大小 × 第二块的大小
```
所以**不需要真的把那个巨大的拼接矩阵算出来**（那会很占内存），
直接"w1 的大小 × w2 的大小"就行，结果一模一样，还省内存。

### 支持的 LoKr 存储形式（重要）

LoKr 的 w1 / w2 可能有几种存法，本节点都处理了（拼接逻辑跟 ComfyUI/LyCORIS 推理时一致）：
- **整块**：`.lokr_w1` / `.lokr_w2` 直接就是完整的一块。
- **拆成两块**：`.lokr_w1_a × .lokr_w1_b`、`.lokr_w2_a × .lokr_w2_b`，要先乘回去才是完整的。
- **tucker 形式（只有 w2 会用）**：`.lokr_t2` 加上两块，用特定方式拼。
- 另外兼容 `.dora_scale`（DoRA 用的）：原样保留、不缩放。

缩放时：每个模块只把系数乘到 **w1 那一侧的一块**——有整块 `.lokr_w1` 就缩它，
没有就缩拆开后的 `.lokr_w1_a`（缩其中一块，等于缩了乘积）。
**绝不同时缩两块，也绝不碰 w2 那一侧的任何东西。**

算影响力时：必须先把上面这些形式**拼回完整的 w1 / w2** 再算大小。
⚠️ 不能拿拆开的 `.lokr_w2_a`、`.lokr_w2_b` 单独算——因为"两块乘积的大小"不等于
"两块各自大小相乘"，而且只取一块会漏算，导致 block 排序乱掉。
（这个坑在真实文件上确实踩到过，见下面验证记录。）

### 验证状态（已在 3 个真实 Anima LoKr 上验证，不是合成数据、不是"想当然"）

用 torch 加三个真实文件做了完整验证，全部通过。这三个文件覆盖了 LoKr 的几种不同存法：

| 文件 | w2 怎么存的 | alpha | 是否带 DoRA | 高能量段（top） |
|---|---|---|---|---|
| `4x1style_v2_1_epoch48` | 拆成两块 `a×b` | 浮点 32 | 是 | 16,15,17 |
| `nakkar_v2_epoch26` | 拆成两块 `a×b` | 浮点 32 | 否 | 15,16,17 |
| `ReiNa_style_v6_step4000` | 整块 | 整数 8 | 否 | 15,17,18 |

验证的几件事：
- **染色排序正确**：节点算出的每个 block 影响力排序，和"老老实实把等效权重整个拼出来再算"
  得到的标准答案，逐位一致。三个文件高能量段都落在 **14~19 这一带**，
  跟发布版 LoRA 实测的"核心表达段 15-18"吻合。
- **缩放精确**：per_block 模式把 blk16 设成 0.5，重建出的等效权重缩放比正好 0.5000，
  其它 block 不受影响。
- **没有坏数**：三个文件的权重张量里都没有 inf 或 NaN，烘焙导出的文件也没有。
- **alpha 各种类型都稳**：三个文件 alpha 分别是浮点和整数（`ReiNa` 是 64 位整数），
  因为本节点压根不读 alpha，所以不管它是什么类型、是不是异常值，都原样保留、不出问题。

> 📌 修 bug 记录：节点初版算影响力时只认"整块 w2"，遇到前两个"拆成两块"的文件就会退化成
> "只看 w1"，实测排序就错了（比如把 block 25/26 错排到前面）。已修成"先把两块拼回完整 w2
> 再算"，修完排序就对了。这个 bug 就是靠真实文件才发现的，合成数据测不出来。

> ⚠️ 一句必须说清楚的话：**能量大 ≠ 功能强**。上面说的"高能量段在 15-18"，只是说这些 block
> 的权重数值最大，并不代表它们就是控制画风、骨架那些"功能段"。LoKr 是和普通 LoRA 不同的
> 数学拆法，到底哪个 block 调画面的什么，还得靠在 ComfyUI 里固定种子、一个变量一个变量地
> 试出来。本实验版**只验证了"数学算对了"，没验证"出图功能图"**——这是有意为之，见下一节。

## 这个实验版到底是什么定位（重要，请先读这段）

简单说：**这个 LoKr 版本"数学上验证通过了，但没有做出图实测"，所以它会一直保持"实验版"状态，
不会合并进正式发布的节点。**

具体区分一下"验过"和"没验"：

**已经验过的（用 torch 在 3 个真实文件上跑过）：**
- LoKr 的缩放数学是对的——只缩 w1 一侧，等效于整体缩放，不会变成平方。
- 影响力染色的排序是对的——和把权重整个拼出来算的标准答案一致。
- 各种存储形式（整块 / 拆两块 / tucker）、各种 alpha 类型（浮点 / 整数 / 异常的 inf）都不会崩。

**没有验、也不打算在这个版本里验的：**
- 出图效果到底对不对、好不好看。
- LoKr 的"功能地图"——也就是拉某个 block 的滑块，画面具体会变什么。
  这个必须在 ComfyUI 里固定种子、一个变量一个变量试，工作量大，暂时不做。

**为什么这样安排：**
- 发布版那两个节点是给普通 LoRA 用的，已经经过实测、稳定，不动它。
- LoKr 这套留作实验版，谁想用 LoKr 分层可以试，但心里清楚"数学对，功能图未实测"。
- 这样既不耽误想用的人，也不会因为没实测就贸然当成正式功能误导别人。
- 实验分支是用 `try/except` 安全加载的，就算它哪天出错，也完全不影响发布版那两个节点。

补充一个背景：很多现成的 Anima LoKr 还停留在比较老的训练版本（比如 anima p2/p3），
参考价值有限，所以这里也没必要为了凑实测样本去硬跑。保持"验证通过的实验版"这个状态就够了。

## 已知注意事项

- 有些 LoKr 训练时用了个小手段：把某些参数（dim/alpha）设成超大的值交给优化器自己调，
  存成半精度后这些值会溢出成 inf（无穷大）。本节点**完全不读这些参数**，只对 w1 那一块做乘法，
  所以天然绕开了 inf 引发的坏数问题。已验证导出的文件里权重没有坏数。
- 分层时"哪个权重属于哪个 block"的判断逻辑，跟发布版完全一样。
- 参数和发布版节点一样，调参思路、扫描定位的玩法都通用。
- 影响力染色里"不读 alpha"有个边界：目前测的几个文件，alpha 除以 rank 正好是 1、且每个模块都一样，
  所以不影响染色的相对深浅。万一遇到 alpha/rank 不是 1 且每个模块还不同的 LoKr，
  染色的深浅可能略有偏差（但缩放和出图本身仍然是对的，只是颜色深浅不够准）。

## 安装

experimental 子目录会被主 `__init__.py` 自动尝试加载。无需额外操作；
若不想启用实验功能，删除整个 `experimental/` 目录即可，不影响发布版节点。
